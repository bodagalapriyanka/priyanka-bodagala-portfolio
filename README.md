## Use this repo to create your own portfolio

This is a plain static website, so you can reuse it without setting up a frontend framework.

1. **Fork this repository** from GitHub so you have your own copy.
2. **Clone your fork** to your computer.
3. **Replace the personal details** in `index.html`, `README.md`, and `docs/portfolio-data.md`:
   - Name, headline, location, and contact email
   - LinkedIn, GitHub, Google Scholar, ORCID, ResearchGate, and other profile links
   - Roles, education, service, speaking, publications, articles, and media
   - Featured impact metrics and technical stack
   - Projects, project outcomes, tools used, links, and screenshots or visuals
   - Publication links, DOI links, article URLs, project demos, GitHub repos, and portfolio case-study links
   - Preferred contact options, such as email, LinkedIn, GitHub, Calendly, personal website, newsletter, or location
4. **Replace visual assets** in `assets/images/`.
5. **Add optional downloadable files** in `assets/files/`, if you choose to include any:
   - Resume PDF
   - CV PDF
   - LinkedIn profile PDF
   - Project one-pagers or selected public writing samples
6. **Preview the site locally** by opening `index.html` in your browser or running a simple local static server.
7. **Deploy the site** with Vercel, GitHub Pages, Netlify, or another static hosting provider.

## Add resume, LinkedIn PDF, links, and contact details

Use this pattern for optional public files and links:

1. **Resume or CV**
   - Save the file as a PDF, for example `resume.pdf` or `cv.pdf`.
   - Put it in `assets/files/`.
   - Add or update a link in `index.html`, for example `assets/files/resume.pdf`.
   - Only include a resume publicly if you are comfortable with every detail being visible online.

2. **LinkedIn profile PDF**
   - On LinkedIn, open your profile.
   - Use **More** > **Save to PDF**.
   - Rename the file clearly, for example `linkedin-profile.pdf`.
   - Put it in `assets/files/`.
   - Add the link in `index.html` if you want visitors to download it.

3. **Publication and article links**
   - Add DOI links, journal links, Google Scholar links, ResearchGate links, PubMed links, or publisher URLs.
   - Keep titles, author role, venue, and year consistent across `index.html`, `README.md`, and `docs/portfolio-data.md`.

4. **Project links**
   - Add GitHub repo links, demo links, writeups, screenshots, or short case studies.
   - For private or employer-sensitive projects, describe the public-safe outcome without exposing confidential data, private screenshots, credentials, patient data, or internal system details.

5. **Contact and profile URLs**
   - Update email, LinkedIn, GitHub, Google Scholar, ORCID, ResearchGate, personal website, newsletter, Calendly, or other preferred contact links.
   - Search the repo for old emails and URLs before publishing so outdated contact details are not left behind.

## If you are new to GitHub or Vercel

Create these free accounts first:

1. **GitHub account**
   - Go to [github.com](https://github.com/) and create an account.
   - Use GitHub to fork this repo, save your customized website code, and track updates.

2. **Vercel account**
   - Go to [vercel.com](https://vercel.com/) and create an account.
   - Sign in with GitHub so Vercel can import and deploy your portfolio repo.
   - After importing the repo, Vercel will provide a live website URL.

You can also use GitHub Pages, Netlify, or another static hosting service if you do not want to use Vercel.

## Use Codex or Claude Code to customize and publish it

You can use Codex, Claude Code, or a similar coding assistant to do the setup and edits after forking the repo. The same basic workflow applies: inspect first, edit content, preview locally, connect Git, push to GitHub, then deploy.

1. **Fork the repo on GitHub.**
   - Click **Fork** on the original repository.
   - Choose your GitHub account as the destination.

2. **Open Codex or Claude Code in the folder where you want the website.**
   - Ask the assistant: `Clone my forked portfolio repository from GitHub.`
   - Share your fork URL when it asks for it.

3. **Ask the assistant to inspect before editing.**
   - Example prompt: `Inspect the portfolio files first. Do not change anything yet. Tell me where my name, links, projects, publications, and contact sections live.`

4. **Ask the assistant to replace the content.**
   - Example prompt: `Update this portfolio with my personal details, projects, publications, experience, education, links, downloadable public PDFs, and contact email. Keep it public-safe and do not include private files.`

5. **Ask the assistant to connect Git.**
   - Example prompt: `Check the git status and remote. Connect this folder to my GitHub fork if needed.`

6. **Ask the assistant to preview the site.**
   - Example prompt: `Start a local preview and tell me the localhost URL.`

7. **Ask the assistant to commit and push when ready.**
   - Example prompt: `Commit these portfolio updates and push them to my GitHub fork.`

8. **Ask the assistant to deploy.**
   - Example prompt: `Deploy this static portfolio to Vercel and give me the live URL.`

Recommended assistant rule: ask it to inspect first, then edit, then preview, then commit/push, then deploy. That keeps the changes easier to review at each step.
